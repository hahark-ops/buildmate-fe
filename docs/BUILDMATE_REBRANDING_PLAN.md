# Buildmate Rebranding Plan

## 1. 리브랜딩 목표
기존 커뮤니티 프로젝트를 `빌드메이트`로 리브랜딩해, 사이드 프로젝트 팀원을 찾고 공개 질문과 실시간 채팅으로 협업을 시작할 수 있는 목적형 서비스로 전환합니다.

이 리브랜딩의 핵심은 `커뮤니티를 조금 더 예쁘게 바꾸는 것`이 아니라, 현재 구조를 `팀원 모집 서비스`처럼 다시 보이게 만드는 것입니다.

## 2. 제품 포지셔닝
- 서비스명: 빌드메이트
- 제품 카테고리: 팀원 모집 + 협업 시작 서비스
- 핵심 메시지: 혼자 만들다 막히면, 함께 만들 메이트를 찾으세요.
- 서비스 정의: 프로젝트를 함께 만들 사람을 찾고, 댓글과 채팅으로 빠르게 협업 논의를 시작할 수 있는 팀 빌딩 플랫폼

### 중요한 설명 원칙
- `AI 툴`은 이 서비스의 메인 정체성이라기보다, 주요 타깃 사용자를 설명하는 웨지입니다.
- 따라서 제품 설명은 `AI 툴 사용자 전용 서비스`보다 `팀원 모집과 협업 시작에 최적화된 서비스`를 우선으로 둡니다.
- 메인/로그인/회원가입의 1차 카피에서 `AI 툴`은 보조 설명이나 타깃 설명에만 사용하고, 대표 헤드라인과 핵심 CTA는 `팀원 모집`, `질문`, `협업 채팅` 중심으로 유지합니다.

## 3. 핵심 사용자
### 3-1. 프로젝트를 올리는 사용자
- 아이디어는 있지만 혼자 완성하기 어려운 사람
- 역할별 팀원을 구하고 싶은 메이커
- MVP를 빠르게 만들어보고 싶은 사람

### 3-2. 참여할 프로젝트를 찾는 사용자
- 합류할 만한 프로젝트를 찾는 메이커
- 실전 협업 경험을 쌓고 싶은 입문자
- 자신의 역할(프론트/백엔드/디자인/기획)에 맞는 팀을 찾는 사람

### 3-3. 사용자가 겪는 문제
- 프로젝트를 혼자 시작했지만 중간에 막힌다.
- 필요한 역할의 팀원을 어디서 찾아야 할지 막막하다.
- 괜찮은 프로젝트를 찾더라도 관심 표현과 실제 대화 시작이 분리돼 있다.
- 공개 질문, 관심 표시, 실제 협업 논의가 한 흐름으로 이어지지 않는다.

## 4. 현재 기능을 Buildmate 문맥으로 재해석하는 방식
- 게시글: 프로젝트 모집글
- 댓글: 공개 질문 / 참여 문의
- 좋아요: 합류 관심 표시
- DM: 협업 제안과 구체 논의를 시작하는 채널
- unread/read: 협업 메시지 상태 확인
- Web Push: 부재 중 협업 알림 구조

핵심은 `기능 추가`보다 `현재 기능이 팀 모집 서비스의 흐름으로 읽히게 만드는 것`입니다.

## 5. UX 재설계 원칙
### 5-1. 범용 커뮤니티처럼 보이지 않게 만들기
- 자유 게시판 느낌을 줄이고, `팀원 모집 보드`처럼 보여야 합니다.
- 메인 화면에서 사용자는 “이곳은 팀원을 찾는 공간”이라고 즉시 이해해야 합니다.

### 5-2. 모집글이 모집 공고처럼 읽히게 만들기
- 제목, 본문, 버튼, 빈 상태, 댓글 입력 문구가 모두 프로젝트 모집 문맥을 가져야 합니다.
- 이번 1차 리브랜딩에서는 DB 스키마를 바꾸지 않더라도, UI에서 아래 신호가 보이게 유도합니다.
  - 필요한 역할
  - 프로젝트 진행 단계
  - 사용하는 도구/기술
  - 협업 방식
  - 원하는 합류 형태

1차 구현에서 위 신호를 실제 화면에 드러내기 위한 최소 UI 스펙은 아래와 같습니다.
- 작성/수정 화면에서 본문 입력 위에 `프로젝트 소개`, `현재 단계`, `필요 역할`, `사용 도구`, `협업 방식`, `합류 형태`를 적도록 유도하는 고정 helper 또는 starter template을 제공합니다.
- 목록 카드에는 작은 상태 배지(`모집 중`)와 메타 라벨(`관심`, `질문`, `조회`)을 사용해 일반 게시판 카드와 인상을 구분합니다.
- 상세 화면에서는 본문에 `현재 단계:`, `필요 역할:`, `사용 도구:`, `협업 방식:`, `합류 형태:`처럼 적힌 줄이 있으면 상단 key facts 행으로 우선 노출하고, 없으면 본문만 노출합니다.

### 5-3. 공개 반응에서 개인 대화로 자연스럽게 이어지게 만들기
- 댓글: 질문 / 공개 문의
- 좋아요: 관심 표시
- DM: 구체적인 협업 제안과 일정/역할 논의

### 5-4. 프로필이 팀원 판단 자료가 되게 만들기
- 프로필은 단순 회원정보가 아니라 `이 사람과 함께 일할 수 있을까`를 판단하는 표면이어야 합니다.
- 이번 단계에서 실제 필드 추가가 어렵다면, 최소한 카피와 구조는 메이커 프로필처럼 보이게 조정합니다.

## 6. 화면별 재설계 방향
### 6-1. 메인 페이지 (`index.html`)
목표: 일반 게시판이 아니라 `프로젝트 모집 보드`처럼 보이게 만들기

필수 변경:
- 로고/서비스명 교체
- 메인 헤드라인/서브카피 교체
- `게시글 작성` 버튼을 `프로젝트 모집하기`로 변경
- 메인 보조 설명에 `합류할 프로젝트를 둘러보세요` 관점을 명시
- 카드 상단에 `모집 중` 배지 또는 이에 준하는 모집 신호를 추가
- 카드 메타 라벨을 `관심 / 질문 / 조회`로 교체
- 목록 카드의 문맥을 모집글처럼 읽히게 조정
- 빈 상태를 `첫 모집글을 올려보세요`로 교체

### 6-2. 로그인 / 회원가입 (`login.html`, `signup.html`)
목표: 계정 생성이 곧 프로젝트 협업 시작이라는 인상을 주기

필수 변경:
- 제목/설명 문구를 협업 시작 문맥으로 교체
- 회원가입 성공 문구는 로그인 이동을 유지하되 서비스 맥락을 반영

### 6-3. 모집글 작성 / 수정 (`post_write.html`, `post_edit.html`)
목표: 자유글 작성이 아니라 `프로젝트 모집글 작성/수정`처럼 보이게 만들기

필수 변경:
- 페이지 제목 변경
- 제목 placeholder 변경
- 본문 안내 문구 변경
- 본문 입력 전에 `이 정보는 꼭 적어주세요` 성격의 구조 가이드를 제공
- 역할 / 현재 상태 / 협업 방식 / 원하는 메이트를 적도록 유도

### 6-4. 모집글 상세 (`post_detail.html`)
목표: 프로젝트 설명과 초기 상호작용의 중심 화면처럼 보이게 만들기

필수 변경:
- `좋아요수`를 `관심 수`로 변경
- 댓글 placeholder와 버튼을 공개 질문/문의 맥락으로 변경
- 본문에 구조 신호가 포함되어 있으면 상단 key facts로 먼저 보여주기
- 작성자 카드 CTA를 `협업 제안하기`로 변경
- 댓글 수정/삭제 상태 문구도 전체 맥락과 충돌하지 않게 정리

### 6-5. DM (`dm.html`, `js/dm.js`)
목표: 단순 채팅이 아니라 `협업 시작 채널`로 느껴지게 만들기

필수 변경:
- 공용 헤더와 패널의 대표 용어를 `협업 채팅`으로 통일
- 상태 문구를 실시간 협업 논의 중심으로 교체
- empty state와 input placeholder를 협업 문맥으로 교체
- push 힌트도 프로젝트 대화 중심으로 표현
- 방 상태, room preview fallback, 연결 상태, 재연결 상태 문구까지 포함해 정리

### 6-6. 프로필 / 비밀번호 / 드롭다운 (`profile.html`, `password.html`, 공용 헤더)
목표: 단순 설정 화면이 아니라 `메이커 프로필 + 계정 보안`처럼 보이게 만들기

필수 변경:
- `회원정보수정` -> `메이커 프로필`
- `비밀번호수정` -> `계정 보안`
- 프로필 화면 제목/버튼/탈퇴 문구 정리
- 현재 `수정하기`와 `수정완료`로 나뉜 버튼 의미가 충돌하지 않게, 저장 버튼과 이동 버튼의 역할을 카피에서 분명히 분리
- 이동 버튼은 특정 랜딩 목적지를 강제하기보다, 사용자가 보고 있던 흐름으로 자연스럽게 복귀하는 의미를 유지
- 프로필 카드 캡션과 CTA도 협업 문맥으로 정리

## 7. 정보 구조에서 반드시 드러나야 할 신호
이번 리브랜딩이 단순 카피 교체로 보이지 않으려면, 아래 정보가 적어도 시각적으로 암시돼야 합니다.
- 이 프로젝트는 무엇을 만드는가?
- 어떤 역할이 필요한가?
- 현재 어느 단계인가?
- 누가 합류하면 좋은가?
- 공개 질문을 남길지, DM으로 바로 제안할지 사용자가 판단할 수 있는가?

1차 구현에서는 데이터 구조를 바꾸지 않더라도, 아래 요소를 UI 문구/레이아웃으로 유도합니다.
- 모집글 제목의 목적성 강화
- 본문 안내 문구 강화
- 카드/상세에서 역할/상태를 적게 유도하는 카피 사용
- 본문 접두어(`현재 단계:`, `필요 역할:`, `사용 도구:`, `협업 방식:`, `합류 형태:`)를 key facts로 노출하는 규칙 정의
- CTA 문구를 `잡담`이 아니라 `질문 -> 관심 -> 협업 제안` 흐름으로 통일

## 8. 단계별 구현 계획
### 0단계: 인벤토리와 공통 셸 정리
- 현재 문자열 인벤토리 수집
- 헤더/로고/드롭다운/공통 CTA 정리
- 테스트에서 카피에 의존하는 selector 확인

### 1단계: 핵심 카피 교체
- 서비스명 교체
- 메인 카피 교체
- 로그인/회원가입/프로필/DM 카피 교체
- README 소개문 교체

### 2단계: 모집글 문맥 강화
- 메인 페이지를 모집 보드처럼 보이게 조정
- 작성/수정/상세 화면을 모집글 중심으로 재정의
- 댓글/좋아요/DM 의미를 팀 빌딩 흐름에 맞게 정리

### 3단계: 시각 리브랜딩
- 색상, 배지, 카드, 버튼 인상 조정
- 원본 과제 느낌이 강한 영역부터 우선 변경

### 4단계: README / 영상 / 제출물 정리
- README를 Buildmate 기준으로 재작성
- 영상 데모를 `모집글 작성 -> 공개 질문 -> DM -> unread/read` 흐름으로 재구성
- 구글폼 제출용 제목/한 줄 설명/영상 소개 문구 정리

## 11. 완료 기준
- 첫 화면만 봐도 원본 과제와 구별된다.
- 사용자가 “여기서 팀원을 찾고 바로 협업을 시작할 수 있다”는 점을 즉시 이해한다.
- 프로젝트를 올리는 사람과 합류할 프로젝트를 찾는 사람 모두의 흐름이 보인다.
- 게시글/댓글/관심/DM이 하나의 팀 빌딩 흐름으로 자연스럽게 이어진다.
- README와 영상 설명이 모두 Buildmate 기준으로 일관된다.
- 기존 기술 강점(DM, Redis, CI/CD, k3s, 테스트코드)은 그대로 살아 있다.
